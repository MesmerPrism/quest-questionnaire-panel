Quest Questionnaire Panel

A reusable Quest panel for questionnaire screens in XR studies.

This project is a native Android panel app for Quest. It shows 2D questionnaire screens, collects answers, writes a JSON result back to the app that launched it, and then closes itself. The Big Red Button Unity project is the first caller, but the contract is meant to work for other native, Unity, or OpenXR apps too.

Panel project Native Quest 2D questionnaire app
First caller Big Red Button Unity study app
Result path caller-owned content URI plus callback

Start Here

Read the project as two cooperating apps, not as one Unity project with a form bolted into it.

1. The XR app owns the study.

The caller keeps the 3D scene, condition timing, participant session, and any app-specific logic. In the BRB case, that means Unity owns the room, the button, condition audio, press counting, and final physical interaction through the Unity launcher.

2. The panel owns the questionnaire moment.

When the caller needs form-like input, it launches this native 2D panel with a request. The panel renders only the requested stage or sequence, plays panel-owned questionnaire prompts, validates required answers, and submits a result from QuestionnaireActivity.

3. The result goes back through Android IPC.

The caller creates a private result file, exposes exactly that file as a writable content:// URI, and gives the panel a one-shot broadcast PendingIntent. The panel writes JSON to the URI, sends the callback as a signal, and finishes only its own activity. The full rule set is in the Intent contract.

How BRB Uses The Panel

The Big Red Button Unity app is the current integration target. It drives the panel by choosing the questionnaire stage, creating the request, and validating the returned JSON. The current reference is the pushed BRB branch codex/brb-questionnaire-panel-bridge.

BRB Unity app owns

  • The 3D Big Red Button scene and press animation.
  • Condition sessions, condition instruction audio, and timing through QuestVrInputManager.
  • Button press counting and final physical interaction through BigRedButtonWorldPressCounter.
  • The session map that links a panel request to the study run.
  • The result URI, callback receiver, and result validation.

Questionnaire panel owns

  • Foreground 2D Quest questionnaire screens in QuestionnaireActivity.
  • Language, demographics, prior experience, and post-condition forms.
  • Panel-owned questionnaire narration and UI feedback audio.
  • Final confirmation and return prompts.
  • Result JSON writing to the caller-granted URI using the shared result model.
BRB moment Panel sequence Plain-language purpose
Before the first condition language_select -> demographics -> prior_experience Set language and collect starting context before the study begins.
After condition 1 post_condition:pictographic -> post_condition:presence_questionnaire -> post_condition:lost_opportunity Ask how the button felt and whether the participant noticed a lost opportunity.
After condition 2 post_condition:pictographic -> post_condition:presence_questionnaire Repeat the post-condition measures without the first-condition-only prompt.
End of the study final:end_confirmation -> final:extra_presses_prompt -> complete:export_summary Ask the 1-to-10 end confirmation, then return to Unity for any physical final presses.

Use It From Another XR App

BRB is the first integration, not a special transport. Any cooperating XR app can use the same launch and result pattern. The reusable rules are documented in the handoff contract and the Intent contract.

What your app must provide

  1. Create a session id, request id, and random nonce.
  2. Persist pending session state before launching the panel.
  3. Create a private result file and expose only that file as a writable content:// URI.
  4. Build request JSON with protocol_version, study_id, schema_id, open_stage, and screen_sequence.
  5. Launch the panel with an explicit component and write permission for the result URI.
  6. Receive the callback, read the JSON from your own file, and validate request id, nonce, schema, status, and answer shape.

What this avoids

The product path does not need ADB, WiFi debugging, public shared storage, MediaStore answer exchange, Termux file drops, package killing, force-stop cleanup, or overlay tricks. Those can be lab tools, but they are not the app-to-app contract.

For a concrete Unity caller, read BRB's result provider, result store, and return receiver.

Contract In One Pass

The launch Intent carries identity, request JSON, a writable result URI, and a callback. The callback is only a signal; answers stay in the result JSON. The canonical definitions live in the panel repo's Kotlin contract, request schema, and result schema.

Required extra Type Why it exists
session_id string Caller session identifier.
request_id string Opaque per-request id used to match the result.
request_nonce string Random value that binds launch and result.
request_json string JSON matching quest.questionnaire.v1.request.
result_uri parcelable Uri Caller-owned writable content:// URI.
return_to_caller PendingIntent One-shot broadcast callback sent after JSON is written.

Commands For Developers

These ADB commands are development and capture helpers. They are useful for repeatable headset checks, but the product integration is still the explicit Intent, result URI, and callback contract above.

Run a Unity-side button command

Use this to prove the BRB Unity runtime is receiving command extras.

adb shell am start -W \
  -n org.thebigredbuttoninstitute.app/com.unity3d.player.UnityPlayerGameActivity \
  --es brb.runtimeCommand press_button

Open the initial panel sequence from Unity

Unity prepares the request and launches the panel route.

adb shell am start -W \
  -n org.thebigredbuttoninstitute.app/com.unity3d.player.UnityPlayerGameActivity \
  --ez brb.questionnaireOpen true \
  --es brb.questionnaireTrigger initial

Open a post-condition sequence

Use the numbered trigger after condition 1 or condition 2.

adb shell am start -W \
  -n org.thebigredbuttoninstitute.app/com.unity3d.player.UnityPlayerGameActivity \
  --ez brb.questionnaireOpen true \
  --es brb.questionnaireTrigger post_condition_1

Drive final panel input at human pace

Debug scripts are for validation clips and regression checks.

adb shell am start -W \
  -n org.thebigredbuttoninstitute.app/com.unity3d.player.UnityPlayerGameActivity \
  --ez brb.questionnaireOpen true \
  --es brb.questionnaireTrigger final \
  --es brb.questionnaireCommandScript "final:1,next,submit" \
  --ei brb.questionnaireCommandIntervalMs 1400

What The Pieces Look Like

These clips show the Unity side and the panel side separately. The point is not that ADB is required; it is that each action can be triggered and validated repeatably while the integration is being built.

Unity: blink the button

The BRB app can run a timed blink on the 3D button cap. This stays on the Unity side because it belongs to the study world.

Unity: press animation and counter

A runtime command plays the press animation and increments the world-space counter. The panel does not replace this physical interaction.

Panel: launch from the caller

Unity opens the native 2D panel with an explicit Android Intent while the study scene remains the caller-owned context.

Panel: move through screens

The panel renders the requested BRB sequence, gathers answers, and enforces required inputs before moving forward.

Panel: submit and close

On submit, the panel writes the result JSON to the granted URI, sends the callback, and finishes only the panel activity.

End-To-End Flow

This is the whole handoff in the order a caller should implement it.

  1. Caller prepares request Persist pending state, create request id, nonce, and private result file.
  2. Caller launches panel Use an explicit component, request JSON, and write-granted result URI.
  3. Panel collects answers Render only the requested stage sequence in the native 2D UI.
  4. Panel writes result Write JSON, close the stream, send the caller callback, then finish.
  5. Caller validates Check protocol, request id, nonce, schema, status, stage, and answer shape.
